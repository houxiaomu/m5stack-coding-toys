# Firmware / hardware gotchas (ESP32-S3 / CoreS3)

Hardware-only failure modes found on real CoreS3 SE:

- **Serial RX buffer (the big one):** ESP32-S3 HWCDC RX FIFO defaults to **256B**. Git-enriched status frames exceed 700B and arrived truncated → decode failed → device silently stuck on Waiting (small frames <256B worked, masking it). Fix: `Serial.setRxBufferSize(4096)` before `begin` in the CoreS3 transport + grew the NDJSON framer. **When adding frame fields or supporting a new board, keep status frames within the device RX buffer or bump it.**
- **Native USB-CDC ignores nominal baud:** CoreS3 native USB-CDC transfers run at full USB speed regardless of the configured baud (153.6KB in ~754ms) — don't reason about throughput from the baud setting.
- **Reset paths differ:** CoreS3 SE's default esptool RTS hard reset (`Hard resetting via RTS pin...`) is unreliable and can leave the board in ROM/stub mode. After flashing, `esptool.py --after watchdog_reset run` has been hardware-verified to boot the app and bring the daemon back to `Connected` without a manual RESET. This does **not** enter download mode; long-press RESET is still required before flashing. Flash via the `m5stack-cores3-bring-up` skill (`.claude/skills/m5stack-cores3-bring-up/`) — UiFlow2 blocks first flash, USB_MODE matters.
- **Orphan daemons grab the serial port:** a stale daemon from a prior session can linger (PPID 1, old dist code) and silently grab the port, causing a phantom "Connected" state. Catch with `ps -ax | grep daemon/dist/main.js`. The singleton / idle-exit work (see [daemon-singleton-and-install.md](daemon-singleton-and-install.md)) largely fixes this.
- **Gated serial debug:** `-DM5CT_DEBUG` / `cores3-se-debug` env + `M5CT_DBG` macro is a dev-only bring-up aid, off by default; the daemon skips non-`{` serial lines so debug output doesn't spam its log.

## Waveshare round AMOLED (CO5300, `firmware-idf/waveshare-amoled`)

Animation / frame-rate findings on the real 466×466 CO5300 QSPI panel (ESP-IDF + LVGL 9 + `esp_lvgl_adapter`, partial-buffer mode):

- **Don't repaint the whole page every tick — it caps you at ~10 FPS.** The UI tick called `refresh_live()` every 80 ms; any flex-child label whose size changed (or just re-running the setters) re-flowed the full-screen flex page, invalidating the **entire 466×466** screen. Each full-screen partial-flush is a ~90 ms QSPI transfer, so the spinner was stuck at ~9–12 FPS **regardless of the animation** (changing arc→dot, `-O2`, etc. all did nothing). Fix: in steady state, **skip the page refresh entirely** (gate on a real data change / view transition) and let the spinner animate on its own via `lv_anim`; and **pin changing labels to a fixed width** (`label_fixed()` — fixed width + `LV_TEXT_ALIGN_CENTER` + `LV_LABEL_LONG_CLIP`) so a text update repaints only that label, never the page. Result: invalidation drops from 462×462 to a small rect, per-frame work 97 ms → ~3 ms, **FPS 9 → 62**. The panel's real ceiling is ~60 FPS (`CONFIG_LV_DEF_REFR_PERIOD=16`), **not** 10 — the 10 was self-inflicted.
- **CO5300 partial writes must be even-aligned, or moving objects leave a trail.** The controller only accepts partial draw windows on even pixel boundaries. LVGL's invalidation rects (e.g. a spinner dot at `cos/sin`-derived odd coords) were sent unaligned, so the vacated old position was never overwritten → a ring of stale dots on the panel. Fix: an `LV_EVENT_INVALIDATE_AREA` rounder (`area_rounder_cb`) that snaps every area to an even origin + even width/height (`x1 &= ~1; y1 &= ~1; x2 |= 1; y2 |= 1`).
- **`lv_snapshot` (the host screenshot path) cannot reveal panel partial-refresh trails** — it re-renders the whole screen into a fresh buffer, so it always looks clean even when the physical panel shows stale pixels. Trail/ghosting bugs must be confirmed on the real device, not from a screenshot.
- **Build `-O2`, not the `-Og` default.** LVGL renders in software (no GPU on S3); `CONFIG_COMPILER_OPTIMIZATION_PERF=y` is in `sdkconfig.defaults`. (On its own it barely moved FPS here because the bottleneck was the full-screen invalidation above — but it's the right baseline and ESP-FAQ's first LCD recommendation.)
- **Diagnosing FPS on-device:** a temporary `SHOW_FPS` build flag in `ui.c` drives an on-screen `N FPS` label (counts `LV_EVENT_RENDER_READY`); add an `LV_EVENT_INVALIDATE_AREA` probe that reports the largest invalidated rect to catch "something repaints the whole screen each frame". Round screen → corner overlays (LVGL's built-in perf monitor) are clipped off the circle; put diagnostics inside the content area.
- **Never rotate a widget with `transform_rotation` on this panel — it hard-hangs the device.** Spinning a session-card icon via `lv_obj_set_style_transform_rotation` (label rotation) froze the firmware so hard that even esptool's USB auto-reset stopped responding (`Failed to connect: No serial data received`) — recovery required a **manual USB replug** (or BOOT+RST into download mode) before it could be reflashed. LVGL widget/label transform needs a rendered intermediate layer, which this 50px partial-buffer + software (no-GPU) renderer can't satisfy. **For rotation use `arc` + `lv_anim` instead** (a fixed-length indicator arc rotated by an lv_anim, like the live ring — no transform layer), or sidestep motion entirely with an **opacity pulse** (breathing) on a solid shape. Both are confirmed safe; label/widget transform is not.
